## 💡 背景・課題 (Why)
カラオケに行った際、「次、何を歌おう？」と悩み、履歴を探す時間を無駄にしてしまう経験から着想を得ました。
**「選曲の迷い」をなくし、歌う時間を最大化する**ために、自分の聴取履歴から瞬時にセットリストを作れるアプリを開発しました。

## 📱 主な機能 (Features)
1.  **Spotify連携によるパーソナライズ**
    * OAuth認証を用いてログインし、「最近聴いた曲」「お気に入り」「おすすめ曲」を自動取得して表示します。
2.  **直感的なプレイリスト作成**
    * 気になった曲を「リストに追加」ボタン一つでストック。カラオケ店での入力をスムーズにするための「セットリスト」を瞬時に構築できます。
3.  **メモ・スコア機能**
    * 曲ごとに「採点結果」、「歌った感想」を記録可能。次回のカラオケに活かせるデータベースになります。

## 🛠 使用技術 (Tech Stack)
### Frontend
* **Language**: JavaScript
* **Framework**: React (Vite)
* **Styling**: CSS (Neon UI Design)

### Backend
* **Language**: Python 3
* **Framework**: Flask
* **Libraries**: Spotipy (Spotify API wrapper), Flask-CORS
* **Authentication**: OAuth 2.0 (Authorization Code Flow)

## 🧑‍💻 私の担当箇所 (Role)
**ポジション：システム統合・フロントエンド設計・Git運用管理**

チーム開発初心者のみの構成であったため、技術的な実装に加え、開発が破綻しないための「ルール作り」を率先して担当しました。

1.  **開発フローの自律的構築（Git管理）**
    * **運用の徹底**: 初めてのバージョン管理に対し、独学で管理手法を学びながら、以下のルールを定義・遵守させました。
        * `main`ブランチへの直接マージの禁止。
        * 機能単位でのブランチ命名規則（feature/機能名など）の策定。
        * **同一ファイルへの同時アクセス制限**: 競合を未然に防ぐため、作業範囲をコンポーネント単位で排他的に割り振る体制を構築しました。
    * **失敗からの学び**: 開発当初は明確なルールがなくコンフリクトが多発しましたが、これを機に「より細粒度なブランチ分割」と「こまめなコミット」の重要性を痛感。現場でフローを修正し続け、最終的には円滑なマージ作業を実現しました。

2.  **フロントエンドのモジュール設計**
    * **コンポーネント分割の徹底**: 開発速度と保守性を両立させるため、Reactのコンポーネント単位でのファイル分割を徹底しました。これによりメンバー間の作業分担を明確化しました。

3.  **フロントエンドとバックエンドのAPI連携**
    * ReactとFlask間のデータ通信を実装。JSONレスポンスの形式や通信フローの可視化を行い、フロント・バックエンドの「つなぎ込み」を完遂しました。

## 🔥 こだわった点・苦労した点 (Challenges)
### 1. 開発スピードと技術理解の両立における反省
バックエンド（Flask）はチーム全員が未経験だったため、48時間という限られた時間内でアプリを完成させることを優先し、プログラムの基礎部分の実装にAIを活用しました。

* **開発の効率化**: AIのサポートを受けることで、未経験の技術を使いながらもスピード感を持って形にすることができました。
* **反省と今後の課題**: 一方で、AIが生成したコードに頼りすぎてしまったことで、内部の仕組みを自分自身で深く理解しきれなかった部分があると感じています。「動くものを作る」だけでなく、使った技術をしっかり自分の血肉にすることがエンジニアとして大切だと痛感しました。今後はAIを単なるツールとしてではなく、自分の理解を深めるための「補助」として適切に使いこなせるようになりたいです。

### 2. 「データは来ているのに表示されない」バグの特定と解決
デバッグ時、ブラウザの検証ツールとサーバーログを突き合わせ、データの型（ArrayかObjectか）や非同期通信の完了タイミングを確認しました。結果、Reactのステート更新とAPIレスポンスの不整合を特定し、コードを修正することで解決しました。

### 3. 「カラオケボックス」を再現したUIデザイン
アプリ全体をダークモード×ネオンカラーで統一。「開いた瞬間にテンションが上がる」没入感のあるデザインを目指しました。

## 🚀 今後の課題と振り返り (Reflections)
* **設計フェーズの重要性**: コーディング前に「Git運用ルール（コミット粒度、ブランチの切り方）」を詳細に詰めることが、開発後半の生産性を左右することを痛感しました。
* **Git運用の高度化**: 今回の経験を教訓に、今後はプルリクエストベースのコードレビューを徹底し、メンテナンス性の高いリポジトリ管理を心がけます。
* **エラーハンドリングの強化**: API制限（Rate Limit）への対応など、より堅牢なシステムの構築を目指します。
* **このサービスはモバイルアプリであるともっと輝けると思うので、バックエンドでのAIの使いすぎた反省と、gitのバージョン管理の反省を生かして開発を続けていきたいと思います
