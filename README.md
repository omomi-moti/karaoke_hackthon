## 💡 背景・課題 (Why)
カラオケに行った際、「次、何を歌おう？」と悩み、履歴を探す時間を無駄にしてしまう経験から着想を得ました。
**「選曲の迷い」をなくし、歌う時間を最大化する**ために、自分の聴取履歴から瞬時にセットリストを作れるアプリを開発しました。

## 📱 主な機能 (Features)
1.  **Spotify連携によるパーソナライズ**
    * OAuth認証を用いてログインし、「最近聴いた曲」「お気に入り」「おすすめ曲」を自動取得して表示します。
2.  **直感的なプレイリスト作成**
    * 気になった曲を「リストに追加」ボタン一つでストック。カラオケ店での入力をスムーズにするための「セットリスト」を瞬時に構築できます。
3.  **メモ・スコア機能**
    * 曲ごとに「歌いやすかったキー」や「採点結果」、「歌った感想」を記録可能。次回のカラオケに活かせるデータベースになります。

## 🛠 使用技術 (Tech Stack)
### Frontend
* **Language**: JavaScript
* **Framework**: React (Vite)
* **Styling**: CSS (Neon UI Design)

### Backend
* **Language**: Python 3
* **Framework**: Flask
* **Libraries**: Spotipy (Spotify API wrapper), Flask-CORS
* **Authentication**: OAuth 2.0 (Authorization Code Flow)

## 🧑‍💻 私の担当箇所 (Role)
**ポジション：Git管理・フロントエンド/バックエンド連携**

チーム開発において、開発環境の整備と、異なる技術（ReactとFlask）を繋ぎこむ「統合」作業を担当しました。

1.  **フロントエンドとバックエンドのAPI連携**

2.  **Git/リポジトリ管理**
    * チーム全体のソースコード管理を担当。コンフリクトの解消やマージ作業を行い、開発がスムーズに進むよう環境を整えました。

## 🔥 こだわった点・苦労した点 (Challenges)
### 1. 「データは来ているのに表示されない」バグの特定と解決
開発中、バックエンド側のログではSpotifyからデータが取得できているにも関わらず、フロントエンド側では何も表示されない（真っ白になる）という重大な問題が発生しました。
どこでデータが途切れているのかを特定するために、ブラウザの検証ツールとサーバーログを突き合わせながら地道にデバッグを行いました。結果、データの受け渡し部分の不整合（型や通信タイミングのズレなど）を特定し、チームメンバーと協力して解決に導きました。

### 2. 「カラオケボックス」を再現したUIデザイン
（チームで協力し）アプリ全体をダークモード×ネオンカラーで統一しました。
単なる管理ツールではなく、「開いた瞬間にテンションが上がる」デザインを目指し、没入感を高めています。

## 🚀 今後の課題と振り返り (Reflections)
初めてのハッカソンで「動くものを作る」ことに注力した結果、多くの学びと課題が得られました。

* **チーム開発における役割分担の難しさ**:
    * 開発当初、「誰がどこまで実装するか」の境界線が曖昧だったため、作業が重複したり、逆に連携部分（APIのレスポンス形式など）で認識のズレが生じる場面がありました。
    * 「走りながら考える」だけでなく、最初にインターフェースの仕様や分担を明確に決めておくことの重要性を痛感しました。
* **Git運用の改善**:
    * 開発スピードを優先したため、コミット粒度が大きくなりがちでした。今後は機能単位でのブランチ運用や、プルリクエストベースの開発を徹底し、履歴をきれいに保つ運用を心がけたいです。
* **エラーハンドリングの強化**:
    * 現在は基本的なエラー処理のみですが、API制限（Rate Limit）にかかった際の待機処理などを追加し、より堅牢なシステムにしていきたいと考えています。
