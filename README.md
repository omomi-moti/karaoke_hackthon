## 💡 背景・課題 (Why)
カラオケに行った際、「次、何を歌おう？」と悩み、履歴を探す時間を無駄にしてしまう経験から着想を得ました。
**「選曲の迷い」をなくし、歌う時間を最大化する**ために、自分の聴取履歴から瞬時にセットリストを作れるアプリを開発しました。

## 📱 主な機能 (Features)
1.  **Spotify連携によるパーソナライズ**
    * OAuth認証を用いてログインし、「最近聴いた曲」「お気に入り」「おすすめ曲」を自動取得して表示します。
2.  **直感的なプレイリスト作成**
    * 気になった曲を「リストに追加」ボタン一つでストック。カラオケ店での入力をスムーズにするための「セットリスト」を瞬時に構築できます。
3.  **メモ・スコア機能**
    * 曲ごとに「採点結果」、「歌った感想」を記録可能。次回のカラオケに活かせるデータベースになります。

## 🛠 使用技術 (Tech Stack)
### Frontend
* **Language**: JavaScript
* **Framework**: React (Vite)
* **Styling**: CSS (Neon UI Design)

### Backend
* **Language**: Python 3
* **Framework**: Flask
* **Libraries**: Spotipy (Spotify API wrapper), Flask-CORS
* **Authentication**: OAuth 2.0 (Authorization Code Flow)

## 🔥 こだわった点・苦労した点 (Challenges)
### 1. 実装スピードを最優先したAI活用とリソース配分
今回、バックエンドの技術（Flask）は私にとって未経験の領域でした。短期間のハッカソンで導入するため、全ての文法を詳細に学習するのではなく、**「期間内に動くものを作り上げるスピード感」**を最優先しました。

* **AIへの委任と実装への集中**:
    複雑な処理ロジックや構文の生成はAIに任せることで、コーディングにかかる時間を大幅に短縮しました。
* **チーム開発への貢献**:
    生成されたコードが入出力（インターフェース）として正しいかどうかの確認に注力し、浮いた時間を「フロントエンドとの連携」や「バグの検証」に充てました。
    これにより、未経験の技術につまづくことなく、チーム全体のゴールである「プロダクトの完成」に貢献することができました。

### 2. 「データは来ているのに表示されない」バグの特定と解決
開発中、バックエンド側のログではデータが取得できているにも関わらず、フロントエンド側では何も表示されない（真っ白になる）問題が発生しました。
どこでデータが途切れているのかを特定するために、ブラウザの検証ツールとサーバーログを突き合わせながらデバッグを行いました。
結果、データの受け渡し部分の不整合（型定義の相違や、非同期処理のタイミングのズレ）を特定し、コードを修正することで解決しました。

### 3. 「カラオケボックス」を再現したUIデザイン
アプリ全体をダークモード×ネオンカラーで統一しました。単なる管理ツールではなく、「開いた瞬間にテンションが上がる」デザインを目指し、没入感を高めています。

## 🚀 今後の課題と振り返り (Reflections)
初めてのハッカソンで「動くものを作る」ことに注力した結果、多くの学びが得られました。次は以下の点を改善し、より品質の高い開発を目指します。

* **設計フェーズの重要性**:
    * 開発当初、「誰がどこまで実装するか」の境界線が曖昧だったため、APIのレスポンス形式などで認識のズレが生じました。次はコーディング前に簡単なAPI仕様書を作成し、インターフェースを明確にしてから実装に入りたいと考えています。
* **Git運用の高度化**:
    * 開発スピードを優先したため、コミット粒度が大きくなりがちでした。今後は機能単位でのブランチ運用やプルリクエストレビューを徹底し、履歴をきれいに保つ運用を心がけたいです。
* **エラーハンドリングの強化**:
    * 現在は基本的なエラー処理のみですが、API制限（Rate Limit）にかかった際の待機処理などを追加し、より堅牢なシステムにしていきたいと考えています。
